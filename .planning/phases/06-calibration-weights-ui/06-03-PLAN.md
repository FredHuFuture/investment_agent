<design_decisions>

**1. Vitest snapshot (not Playwright) for locking behavior:**
Phase 4 explicitly deferred Playwright to UI-v2 per `04-03-SUMMARY.md`. Vitest's built-in `toMatchSnapshot` captures the full rendered DOM (data-testids, title attributes, class names, text content) and fails on any future change — cheap to maintain, zero new dependencies. This is the correct granularity for "lock the current contract" without standing up an e2e harness.

**2. What the snapshot captures vs what the operator verifies:**
- **Snapshot captures:** DOM structure, title attributes (the tooltip text), data-testids, class names (including color classes: bg-amber-500/50 / bg-green-500/50 / bg-red-600/80), children text.
- **Operator verifies:** live browser behavior — actual hover tooltips appearing on mouseenter, the daemon log line after toggle, state persistence on page reload.
- Together: if the snapshot passes, the DOM is unchanged; if the operator verifies once, the live behavior matched the expected DOM. Future regressions in DOM shape are caught by CI; live-behavior regressions require a repeat operator run (acceptable for this tier).

**3. Operator scripts exit behavior (mirrors Phase 5 pattern):**
- Exit 0: operator typed "approved" at the prompt
- Exit 2: operator typed "failed" or aborted; script prints a template for the issue to paste into the UAT doc
- `--approved` CLI flag to skip prompt (for CI rerun / documentation regeneration)
- Scripts NEVER themselves start uvicorn or the frontend — they print the exact commands the operator runs (same pattern as `scripts/verify_close_03_daemon_pid.py`). Exception: the scripts MAY start the backend if `--auto-start` is passed; safer default is print-only.

**4. UAT doc edit format (precise diff):**
Each of the three items needs:
```markdown
**result:** resolved (2026-04-23 — verified via operator run + Vitest snapshot test {path})
**evidence:**
  - Snapshot test: {path}
  - Operator script: scripts/verify_close_0X_{name}.py
  - Verification date: 2026-04-23
  - Verified by: solo-operator
```

**5. Snapshot file naming + location:**
Files end with `.snapshot.test.tsx` (not `.test.tsx`) to distinguish from regular behavioral tests in the same `__tests__/` dir. Snapshots store in `__snapshots__/` subdirectory (Vitest default). If a component change is intentional, developer runs `npx vitest -u` to regenerate.

**6. No changes to production code:**
This plan touches ZERO production `src/` files. Snapshot tests ARE new files in `__tests__/` dirs but they do not modify anything else. Operator scripts live under `scripts/` (existing directory, Phase 5 convention).

**7. Directory creation:**
The `__snapshots__/` subdirectories are created automatically by Vitest on first run. The executor does NOT need `mkdir -p` for those.
